# Граф KODIK: многоагентный движок принятия решений для Product Discovery

**KODIK** - это прототип циклического многоагентного workflow на базе **LangGraph**. Он нужен для того, чтобы не просто "обсудить идею продукта", а прогнать ее через несколько разных ролей и получить управленческое решение:

- запустить дешевый эксперимент;
- подержать гипотезу и собрать еще сигналы;
- масштабировать;
- выйти из ставки.

Главная мысль проекта: прежде чем тратить месяцы разработки, полезно заставить систему формализовать идею, атаковать ее с разных сторон и проверить, заслуживает ли она следующего инвестиционного шага.

---

## Что делает граф

Граф работает как цепочка узлов, которые последовательно читают и обновляют общее состояние `KodikState`.

1. `intention`
   Формализует исходную идею в продуктовую ставку, перечисляет ключевые допущения и выделяет самое хрупкое место гипотезы.
2. `environment`
   Ищет внешние угрозы: рынок, конкуренция, стоимость привлечения, макрофакторы, операционные ограничения.
3. `destructor`
   Проводит premortem-анализ: исходит из предположения, что проект уже провалился, и пытается объяснить почему.
4. `threshold`
   Сводит качественный анализ к решению: `run_experiment`, `hold`, `scale` или `exit`.
5. `experiment`
   Если данных пока мало, предлагает минимальный обратимый эксперимент.
6. `calibration`
   Обновляет доверие к гипотезе на основе результата эксперимента.
7. `finalize`
   Завершает граф и оставляет финальную запись в журнале.

## Где стоят заглушки

Ниже перечислены конкретные места, где в проекте сейчас не реальная логика, а демонстрационные заглушки.

- `kodik/llm.py -> rag_stub()`
  Настоящего RAG здесь нет. Функция не делает retrieval, не ходит в векторную базу и не ищет документы. Она просто собирает текстовый контекст из уже известных полей `state`.
- `kodik/llm.py -> ask_ollama()`
  Если Ollama недоступна, вместо реального ответа модели используется fallback-текст из `kodik/prompts.md`. То есть граф продолжает работать в демо-режиме даже без LLM.
- `kodik/nodes.py -> destructor_node()`
  В деструкторе нет отдельной специализированной логики, данных или инструмента premortem. Узел просто отправляет промпт в LLM и в случае недоступности модели возвращает заглушечный fallback из `prompts.md`.
- `kodik/nodes.py -> threshold_node()`
  Решение арбитра сейчас rule-based, а не выведено из реальных метрик, вероятностей или калиброванной модели. Логика выбора `run_experiment` / `hold` / `scale` / `exit` пока захардкожена в `if/elif`.
- `kodik/nodes.py -> experiment_node()`
  Эксперимент не собирается в строгую структуру из ответа модели. В `cheapest_experiment` сейчас подставляется фиксированное имя, фиксированная стоимость и фиксированная обратимость, а текст LLM кладется только в `success_signal`.
- `kodik/nodes.py -> calibration_node()`
  Узел не читает реальные результаты эксперимента. Вместо этого он всегда добавляет одну и ту же симулированную запись в `calibration_history` и увеличивает `trust_score` на фиксированные `0.15`.
- `kodik/demo.py -> build_demo_state()`
  Начальное состояние целиком демонстрационное: идея, рынок, страна, размер команды, runway и risk appetite захардкожены.
- `kodik/prompts.md`
  В файле лежат не только промпты, но и fallback-тексты для случаев, когда модель недоступна. Это тоже часть текущего демо-режима.

## Как работают вызовы LLM

Сейчас LLM в проекте используется как текстовый движок для отдельных узлов графа, а не как механизм маршрутизации. Переходы между узлами задаются кодом LangGraph, а модель генерирует текстовые выводы для состояния.

Полный путь вызова выглядит так:

1. Узел в `kodik/nodes.py` выбирает нужные `system` и `user` шаблоны.
2. Шаблоны берутся из `kodik/prompts.md` через `get_prompt()` и `render_prompt()` из `kodik/prompts.py`.
3. В `render_prompt()` в шаблон подставляются данные из текущего `state`.
4. Узел вызывает `ask_ollama(...)` из `kodik/llm.py`.
5. `ask_ollama()` создает `OllamaClient` и отправляет HTTP POST на `http://127.0.0.1:11434/api/generate`.
6. Ollama возвращает обычный текстовый ответ.
7. Этот текст записывается в состояние графа, например в `intention_summary`, `environment_brief`, `destructor_brief` или `threshold_reasoning`.

Какие узлы вызывают модель:

- `intention_node()` генерирует формализацию продуктовой ставки;
- `environment_node()` генерирует анализ внешней среды;
- `destructor_node()` генерирует premortem-разбор;
- `threshold_node()` генерирует текстовое обоснование уже выбранного решения;
- `experiment_node()` генерирует описание сигнала успеха для эксперимента.

Что важно в текущей реализации:

- модель по умолчанию задана в `kodik/models.py` в `OllamaSettings`: `qwen2.5:3b`;
- температура тоже задается там же и сейчас равна `0.2`;
- ответ модели почти везде хранится как обычный текст, без строгого structured output;
- LLM сейчас не принимает решение, в какой узел идти дальше: это делает Python-код;
- если Ollama недоступна, `ask_ollama()` не останавливает граф, а возвращает fallback-текст из `kodik/prompts.md`.

Итог: LLM сейчас отвечает за генерацию текстовых рассуждений внутри узлов, а orchestration, state management и routing живут в Python и LangGraph.

---

## Как устроено состояние

Все узлы общаются не напрямую, а через общий словарь состояния `KodikState`. В нем лежат:

- исходный интент пользователя;
- контекст проекта;
- промежуточные аналитические выводы;
- история калибровки;
- `trust_score`;
- номер итерации;
- `audit_log`.

Это удобная модель для LangGraph: каждый узел возвращает только те поля, которые обновил, а граф сам сливает изменения в общее состояние.

## Логика принятия решения

В `threshold_node` сейчас зашита простая rule-based логика:

- если это нулевая итерация, сначала нужен `run_experiment`;
- если `trust_score >= 0.7`, можно `scale`;
- если лимит итераций еще не исчерпан, но уверенности недостаточно, выбирается `hold`;
- если кругов уже слишком много, решение - `exit`.

Это пока прототипная логика, но архитектурно она показывает, где в системе находится точка принятия решения.

## Ограничения текущего прототипа

Проект хорошо показывает архитектуру графа, но пока не является production-ready системой оценки гипотез. В частности:

- нет настоящего RAG;
- нет отдельного специализированного механизма для premortem в `destructor`;
- нет строгого парсинга ответов модели в структуры;
- нет реальных результатов экспериментов в `calibration`;
- нет data-driven логики принятия решения в `threshold`.

## Стек

- Python 3.10+
- `langgraph`
- локальная **Ollama**
- стандартная библиотека Python для HTTP-вызова модели (`urllib.request`)

`requirements.txt` сейчас минимальный:

```txt
langgraph
```

---

## Быстрый старт

### 1. Установить зависимости

```bash
pip install -r requirements.txt
```

### 2. Поднять Ollama

Нужен локальный сервер Ollama и доступная модель `qwen2.5:3b`.

Пример:

```bash
ollama run qwen2.5:3b
```

Если Ollama не запущена, граф все равно отработает, но будет использовать встроенные fallback-ответы.

### 3. Запустить демо

```bash
python main.py
```

Во время выполнения скрипт:

- проходит по узлам графа;
- печатает обновления после каждого шага;
- показывает итоговое решение;
- выводит финальный `trust_score`.

## Что полезно посмотреть в коде

Если заходить в проект впервые, лучше читать код в таком порядке:

1. `kodik/models.py`
   Здесь лежат `KodikState`, `Decision` и остальные базовые структуры данных.
2. `kodik/llm.py`
   Здесь лежат клиент Ollama, `ask_ollama()` и временный `rag_stub()`.
3. `kodik/prompts.md`
   Здесь хранятся system/user prompt-шаблоны и fallback-тексты для LLM-узлов.
4. `kodik/nodes.py`
   Здесь находится основная бизнес-логика узлов графа.
5. `kodik/graph.py`
   Здесь описана схема переходов между узлами и собирается LangGraph.
6. `kodik/demo.py`
   Здесь живет demo-state и код демонстрационного запуска.
7. `main.py`
   Это тонкая точка входа, которая просто запускает демо.

## Куда развивать дальше

Самые логичные следующие шаги:

- заменить `rag_stub()` на настоящий RAG с векторной базой;
- вынести промпты в отдельные шаблоны или конфигурируемый слой;
- добавить строгий парсинг ответов модели;
- подключить реальные результаты экспериментов вместо заглушки в `calibration_node()`;
- заменить rule-based арбитра в `threshold_node()` на более обоснованную логику.
